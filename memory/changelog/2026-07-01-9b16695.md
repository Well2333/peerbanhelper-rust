# 0.1.1 发布前审核修复

## 做了什么

一轮代码审核后修复三处问题（发布 v0.1.1 前折入本版）：

- **GeoIP 并发下载写竞争（中等 bug）**：后台自动下载循环（`pbh-server/main.rs`）**不持** `geoip_lock`，而手动更新端点（`pbh-web/routes.rs`）持锁；两者又都用**固定**临时文件名 `{file}.tmp`。启动初期后台在重试下载、用户此时点「更新」，会争用同一 tmp 路径 → 目标 `.mmdb` 可能被写成截断/损坏。改为临时文件名带 `pid + 进程内原子序号`（`download.rs` 的 `TMP_SEQ`），各写各的完整文件再各自**原子 rename** 到目标，目标永远是完整文件，杜绝写竞争损坏（多余重复下载无害）。
- **手动「更新」= 强制刷新**：`ensure_databases` 新增 `force` 参数；手动端点传 `force=true`，无视 45 天/存在性门槛始终重下最新库。此前库文件较新（<45 天）时点击为空操作、返回 `changed:false`；现在手动点击必定重新拉取。后台循环保持 `force=false`（按门槛下载）。
- **token 常数时间比较（轻微安全）**：`auth`/`login` 的 bearer/登录 token 校验从 `==` 改为 `ct_eq`（长度先判、再逐字节 OR 累积差异），消除 `==` 短路带来的计时侧信道。
- **clippy 清零**：`selfupdate.rs` 的 `spawn_restart` 从 `#[cfg(test)] mod tests` 之后上移到测试模块之前，消除 `items after a test module` 告警。

## 为什么

用户要求发布前再审一遍。审核（含子代理并发路径分析）发现 GeoIP 双下载者共享固定 tmp 的真实损坏窗口、手动更新在库较新时是空操作、token 非常数时间比较三点，逐一修复。

## 影响的文件 / 范围

- `pbh-geoip/src/download.rs`：`TMP_SEQ` 原子序号 + 唯一临时文件名；`ensure_databases` 新增 `force` 形参。
- `pbh-server/src/main.rs`：后台 GeoIP 循环 `ensure_databases(..., force=false, ...)`。
- `pbh-web/src/routes.rs`：`ct_eq` 常数时间比较用于 auth/login；手动 `geoip_update` 传 `force=true`。
- `pbh-web/src/selfupdate.rs`：`spawn_restart` 移到测试模块之前。
- 版本不变（**0.1.1**，本轮修复折入首个自更新可用版本）。
