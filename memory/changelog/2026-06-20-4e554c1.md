# M4 余下规则模块：AutoRangeBan / MultiDialingBlocker / IdleConnectionDosProtection / PTRBlacklist

补齐 M4（规则模块 7/7）。

## 做了什么

### AutoRangeBan（`pbh-engine/src/auto_range_ban.rs`）
- 同段连坐：peer 所在 /N 段（IPv4 默认 /30、IPv6 默认 /48）内若已有别的 IP 被**规则**封禁，则一并封禁。
- 读 `BanList`，无内部状态；排除 `ban_for_disconnect` 记录；握手 peer 放行。
- 为此给 `BanList` 加 `any_active_ban_in(block) -> Option<IpAddr>`。
- 配置：`ipv4`(30) / `ipv6`(48) / `ban-duration`。

### MultiDialingBlocker（`pbh-rules/src/multi_dialing.rs`）
- 同种子 + 同子网（v4 /24、v6 /56）内去重 IP 数超 `tolerate-num-ipv4/ipv6`(2/5) 判多拨，封后续 IP。
- moka 缓存承载段计数（TTL=`cache-lifespan` 86400s）+ 可选「追猎」(`keep-hunting`/`keep-hunting-time`)。
- 段计数用 `moka::sync::Cache<String, Arc<Mutex<HashSet>>>`，TTL 自动清理（替代上游 Caffeine + 批回调）。

### IdleConnectionDosProtection（`pbh-rules/src/idle_dos.rs`）
- 空闲连接（无传输、进度不变）超 `max-allowed-idle-time`(300000ms) 封禁。
- 3 种保护模式：0=按 peer flags（做种总保护/下载看兴趣位）、1=仅做种、2=总是。
- 速度快路径 + 字节增量 + 进度变化判活动；moka time-to-idle 自动驱逐消失的 peer（替代上游 `BatchMonitor` 钩子清理）。
- `uploaded/downloaded == -1`（下载器不可报告）按 0 增量处理。

### PTRBlacklist（`pbh-engine/src/ptr_blacklist.rs`）
- 反向 DNS（PTR）命中 `RuleSet`（STARTS_WITH/.../REGEX）即封。
- DNS 是网络 I/O，无法在同步 `should_ban` 阻塞 → **「后台预解析 + 同步查缓存」**：首次见某 IP 派发 `tokio::spawn` 异步解析（hickory-resolver）、本轮放行；结果入 moka 缓存，下一轮命中再判定。`inflight` 去重避免重复派发。延后一轮判定对持续 ban wave 可接受。
- 配置：`ptr-rules`（JSON 规则数组）/ `cache-ttl`(3600s) / `ban-duration`。

### 接线
- `build_modules` 新增 `ban_list: &Arc<BanList>` 参数（供 AutoRangeBan）+ 4 模块构建（**默认全部关闭**）+ 数值/布尔配置读取助手。
- `main.rs`：先建 `ban_list` 再 `build_modules`；`web::put_profile`：重建时传 `ban_manager.ban_list()`。

## 为什么
- 用户要求「完成路线图其余目标」。M4 此前 3/7，本次补齐剩余 4 个高价值规则。
- **默认关闭**这 4 个（不照搬上游的默认开）：AutoRangeBan/MultiDialing 会扩大封禁面、易误伤（CGNAT 同段正常用户）；Idle/PTR 需要状态/联网。按需在 profile.yml 开启，避免开箱误封。

## 依赖
- `pbh-rules` 加 `parking_lot`；`pbh-engine` 加 `moka(sync)` + `hickory-resolver`。

## 验证
- `cargo clippy -D warnings` 零告警；`cargo fmt --check` 干净；`cargo test --workspace` **58 单测全绿**（新增 AutoRangeBan 3 + MultiDialing 3 + IdleDos 3 + PTR 2）。
- **真机运行时冒烟**（well404-qb，release 实跑）：PUT profile 启用全部 4 新模块（阈值设为不可触发）→ 热重载返回 `modules:4`、**PtrBlacklist 的 resolver 在 async handler 内成功构建**、服务穿越 17 轮 wave / 检查 472 peer、**无 panic/崩溃**、PTR 后台解析正常、无失控封禁。测后恢复原 profile。

## 注意事项 / 后续
- 这 4 个模块的「真实触发」端到端封禁未在用户真机验证——因 AutoRangeBan/MultiDialing 在公网 swarm 上可能误伤真实用户，不宜在用户实例上贸然触发。逻辑由单测覆盖；集成路径（peer→模块→封禁→下发）与已验证的 PeerIdBlacklist 共用同一 trait/流水线。
- IdleDos 的 `idle-speed-threshold` 语义按「字节增量阈值」近似上游的「速度阈值」（避免时间除法歧义）。
- PTR：首次见到的 IP 延后一轮判定（后台解析）。
