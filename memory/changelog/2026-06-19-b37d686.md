# M1 领域模型 + 规则引擎

## 做了什么
- **pbh-rules 匹配引擎**（重做 `matcher`，新增 `ip_matcher`、`cache`）：
  - `Matcher`：STARTS_WITH/ENDS_WITH/CONTAINS/EQUALS/LENGTH（大小写不敏感）+ REGEX（`regex` crate，大小写敏感）。
  - `RuleSet`：FALSE 短路优先级（精确复刻上游）；`RuleSet::parse(&[String])` 从 JSON 规则解析（method/content/min/max/hit）。
  - `IpMatcher<V>`：CIDR/裸 IP 最长前缀匹配（`ip_network_table`），双栈。
  - `ModuleMatchCache<V>`：`moka` 记忆化，含 `get_or_insert` 与 `get_or_insert_pass_only`（仅缓存 pass，对应上游 readCacheButWritePassOnly）。
- **pbh-engine `BanList`**：双栈前缀 trie + `parking_lot::RwLock`；`ban/unban/get/contains/len/remove_expired/snapshot`；
  携带 `ban_for_disconnect` 等 `BanMetadata`；内存权威，DB 仅快照。
- 依赖：pbh-rules 引入 serde_json/regex/ipnet/ip_network(+table)/moka(sync)/thiserror;pbh-engine 引入 ip_network(+table)/parking_lot。

## 为什么
- M1 是规则模块（M4/M5/M6）与流水线（M3）的共同地基：所有按名/按 IP 的封禁都走这套引擎，BanList 是封禁状态的运行期权威。
- 引擎是「等价性关键」，先写测试（守则第 8 条）。

## 影响的文件 / 范围
- pbh-rules：Cargo + matcher.rs(重写)/ip_matcher.rs(新)/cache.rs(新)/lib.rs。
- pbh-engine：Cargo + lib.rs + ban_list.rs(新)。
