# M3 Ban Wave 引擎 + 默认规则模块（二进制可运行）

## 做了什么
- **pbh-rules**：`RuleFeatureModule` trait + 三个离线规则 `PeerIdBlacklist` / `ClientNameBlacklist` / `AntiVampire`（迅雷 0019 仅做种时封）。
- **pbh-downloader**：`DownloaderManager`（`downloaders.yml` 持久化、list、upsert、remove）。
- **pbh-storage**：`upsert_torrent`（ON CONFLICT info_hash）、`insert_ban_history`、`query_ban_history`、`count_ban_history`。
- **pbh-engine**：
  - `BanManager::run_once`：到期解封 → 每个未暂停下载器(登录→拉 torrents→拉 peers→逐 peer 跑模块合并结果→命中 BAN 即写 BanList + 历史落库 + 收集新增)→下发完整封禁列表;旁路名单(IpMatcher)与已封跳过。
  - `spawn_loop`：固定延迟 `interval`，`AtomicBool` + RAII `WaveGuard` 防重叠（不持锁跨 await）。
  - `build_modules`：从 `profile.yml` 构建模块,缺省启用三规则并带内置默认名单（开箱即拦截常见坏客户端）。
- **pbh-server**：装配 `DownloaderManager` + 模块 + `BanManager` 进 `AppContext`;启动 ban wave;`tokio::signal::ctrl_c` 干净退出。

## 为什么
- 这是把 M1（规则/BanList）与 M2（下载器）接成「真正会封 peer」的核心闭环,也是用户可运行测试的最小成品形态（CLI）。

## 影响的文件 / 范围
- 新增 pbh-rules/module.rs、pbh-downloader/manager.rs、pbh-storage/repo.rs、pbh-engine/{ban_manager,modules}.rs。
- 改 各 lib.rs 导出、pbh-server/{Cargo,context,main}、README（运行/测试说明）、roadmap、test-status。
