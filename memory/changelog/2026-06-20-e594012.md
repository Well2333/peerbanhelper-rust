# fix: BTN submit_bans 400 —— 补全缺失的必填字段

## 问题
真机连真实 BTN（sparkle.ghostchu.com）时,上行 `submit_bans` 被服务端返回 **400 Bad Request**（认证通过,故非凭证问题）。

## 排查
1. 只读拉取真实 config:确认 `submit_bans.pow_captcha=false` → **PoW 非根因**。
2. 对照上游 Java `BtnBan` 的 `@SerializedName`：发现我的 `BtnBan` **漏了 4 个必填字段 + 2 个可空字段** → 服务端字段校验失败。
   （尝试 POST 伪造数据探测被安全分类器正确拦截——不应向真实 BTN 网络提交假数据;改用源码对照,authoritative。）

## 修复
- **`BtnBan` 补全**：必填 `torrent_is_private` / `from_peer_traffic`(=peer_downloaded) / `to_peer_traffic`(=peer_uploaded) / `downloader_progress`;可空 `peer_flag`(=flags) / `structured_data`。`Option` 字段**不加** `skip_serializing_if`,在 None 时输出 JSON `null`,匹配上游 Gson `serializeNulls()`。
- **数据链路补齐**：
  - `record_ban` 现捕获 `peer.uploaded` / `peer.downloaded` / `peer.flags.raw` 写入 `history`（此前为 NULL）。
  - `NewBanHistory` / `insert_ban_history` 增 `peer_uploaded` / `peer_downloaded` / `flags` 列。
  - `query_btn_bans` join torrents 取 `private_torrent`;补 `downloader_progress`;因 17 列超 sqlx 元组 `FromRow` 上限(16),`BtnBanRow` 改 `#[derive(sqlx::FromRow)]` + SQL 列别名。
