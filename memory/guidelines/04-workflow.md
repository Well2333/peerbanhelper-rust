# 工作流程（对接守则，补项目细节）

> 过程总纲见 `memory/最高优先级工作守则.md`（最高优先级）。本文件仅补项目化细节。

## 仓库记忆（两级）

- `memory/guidelines/`（一级）：稳定有约束力的规范。改规范须与触发改动**同次提交**。
- `memory/changelog/`（二级）：每个功能提交一条 `YYYY-MM-DD-<short-hash>.md`，模板见守则第 5 条。
- `memory/test-status/`：`已测记录.md` + `待测报告.md`（守则第 7 条）。
- **不使用运行环境私有记忆**（如 `~/.claude/.../memory`）——跨会话知识一律入库。

## 分支与提交

- 分支前缀：`feat/` `fix/` `chore/` `docs/` `guideline/`。**不在主干直接开发。**
- 合并回主干用 **squash**，一条分支对应主干一个干净提交；合并后删分支。
- **两步提交**：先功能提交取短哈希 → 再写同名 changelog 单独提交（changelog 提交本身极简、不再为它写记录）。
- 提交信息：祈使句、说明「做了什么 + 为什么」；涉及对外行为须**同次**更新文档（守则第 6 条）。

## 远程操作

`git push` / tag / Release / 发包等只在用户**当次明确授权**时执行。本地 commit 可按流程进行，提交后停在本地等指示。

## 自驱

开头一次性问清方向性疑点；之后能自决就自决并记录，**不为「想确认 A/B」停下整个任务**；受阻只搁置那一小块、继续其余，并记入待办/受阻。

## 验证

重要改动后在尽量真实环境实际运行验证（加载/命令/接口/干净退出）；只能人工验证的写入 `待测报告.md`；已覆盖且未改动的不重复测。
