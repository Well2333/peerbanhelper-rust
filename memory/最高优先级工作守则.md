# 通用工作守则（AI 开发协作）

> **本守则为最高优先级，凌驾其所在文件的其余内容。**

## 1. 仓库内长期记忆，两级管理

把跨会话的长期知识写进仓库并纳入版本控制，而非依赖运行环境的私有记忆：

- **一级 `memory/guidelines/`**：稳定、有约束力的规范（编码风格、架构约定、命名、
  提交流程、技术决策）。**最高优先级**；任何改动若触及规范，必须在**同一次提交**内
  同步更新对应文件——这是改动的一部分，不可延后。
- **二级 `memory/changelog/`**：每次提交留一条 `YYYY-MM-DD-<commithash>.md`，记录
  「做了什么 / 为什么 / 影响范围 / 注意事项」，供溯源。
- 冲突时一级优于二级；被证伪的记忆立即更正或删除，不保留误导。

## 2. 提问只在开头，之后自驱到底

接到任务后，先做极短预研（读关键代码 / 查 API），把所有会影响方向的疑点**一次性**
问清。过了开头：

- 凡能合理自决的就自己决定并记录在产出说明里，继续干；
- **禁止**因「想确认一下」「想让用户选 A 还是 B」而停下整个任务；
- 遇到确实无法推进的点，**只搁置那一小块**、继续完成其余，并记下
  「待办 / 受阻：X，原因：Y」；
- 只有当全部能做的都做完、只剩受阻项时，才带着具体问题回来。

打断用户去等一个本可自决的问题，可能白白阻塞数小时——视为严重失职。

## 3. 写远程操作需当次明确授权

`git push`、打 tag、创建 Release、发布包等「写远程 / 对外发布」动作，**只在用户当次
明确要求时执行**。「之前授权过一次」不等于持续授权。本地 commit 可按流程进行，但
提交后停在本地等指示；不确定就先问。

## 4. 分支化开发 + squash 合并

每项工作（功能 / 修复 / 文档 / 规范）先开类型前缀分支（`feat/` `fix/` `chore/`
`docs/`），不在主干直接开发；合并回主干用 squash，使一条分支对应主干上一个干净
提交；合并后删除分支（本地 + 远程）。

## 5. 每次提交都写变更记录（两步提交）

先做功能提交，取其短哈希，再按 diff 撰写 `memory/changelog/YYYY-MM-DD-<hash>.md`
并单独提交（记录提交本身极简、不再为它写记录）。这样文件名哈希始终精确对应所述
提交。改动若涉及规范，在记录里注明同步了哪份 guideline。

变更记录模板：

```markdown
---
date: YYYY-MM-DD
commit: <short-hash>
type: feature | fix | refactor | chore | docs | guideline
scope: <模块 或 仓库范围>
guideline_changed: false   # 改了一级规范则为 true，并在下方说明
---

# <一句话标题>

## 做了什么
- ...

## 为什么
- ...

## 影响的文件 / 范围
- ...

## 注意事项 / 后续
- ...

## 规范同步（仅当 guideline_changed = true）
- 更新了 `memory/guidelines/<file>`：<改了什么、为什么>
```

## 6. 文档与代码同提交

凡改动面向用户的行为（命令、配置项、权限、对外接口、依赖关系），必须在**同一次
提交**内同步更新相关文档（README、参考文档、总表）。文档过时视为任务未完成。

## 7. 编译 / 类型通过 ≠ 可用

重要改动完成后，要在尽可能真实的环境里实际运行验证（加载、命令、对外接口、干净
退出），把只能人工 / 特定环境验证的项写成一份「待测报告」。同时维护「已测记录」：
覆盖的代码自上次验证后未改动就**不重复测试**，尤其不重复劳烦用户做同样的人工验证。

## 8. 测试聚焦可隔离的纯逻辑

把与运行时 / 外部环境强耦合的部分和纯逻辑分离，单元测试只覆盖能在纯净宿主里
实例化的纯逻辑，最大化可测面；其余靠编译期检查 + 真实环境手测。

## 9. 依赖抽象、不依赖具体实现

模块间通过抽象接口 / 服务协作，消费方注入抽象、对提供方无编译期依赖；可选能力用
「可选注入」（拿不到则降级照常工作），保证任意组合都不出问题。
